Leash — the bot demo

Setup + a styled walkthrough of the full chat conversation, end to end.

Leash v1 ships as a Telegram bot, not a mobile app — faster to ship, native to the crypto-AI audience, and the control-tower UX is fundamentally conversational anyway. The backend is the same loop M1 already proved: probe → policy → sign on testnet → verify on ledger → unlock.

01

Setup · 4 steps

Local-only first run. The bot uses polling, so no public webhook or hosting is required — it talks to Telegram from your laptop.

Get a bot token from BotFather

Open Telegram, message @BotFather, run /newbot, pick a name + handle. BotFather will reply with a token like 1234567890:ABCdefGhI…. Save it.

Put the token in .env

From a fresh clone:

cp .env.example .env
# edit .env, add the line:
# TELEGRAM_BOT_TOKEN=1234567890:ABCdefGhI…

Install + run the bot

npm install
npm run bot

The bot boots, auto-funds a fresh merchant + agent wallet from the testnet faucet (first run only — save the printed seeds back into .env for re-runs), and starts polling Telegram.

Open Telegram, find your bot, send /start

The bot's username is whatever you set with BotFather. Send /start and you'll see the welcome message. Then send /task lithium supply chain risks 2026 to fire your first real testnet payment.

SECURITY · KEY DISCIPLINE

Never commit your .env — it's gitignored for a reason. Wallet seeds and bot tokens belong in env vars. The auto-funded seeds printed on first run are testnet only (no real-money risk), but you should still save them privately and not share them.

02

Chat walkthrough · what you'll see

The bot sends one message per agent event. You watch the full loop happen in chat, from probing to tx settled, with a button that opens the real explorer page for that transaction.

What's actually happening

Each message on the right is one event from the same agent loop M1 already ran. The bot is a thin shell — it forwards events to chat and shoves your commands back into the agent.

Verify the proof. Tap View on Explorer on the green message. You'll see a real XRPL testnet transaction with tesSUCCESS, the right amount (1000 drops = 0.001 XRP), the merchant wallet as Destination, and a UTF-8 memo matching the bot's memo: leash-… line.

Notable

  • Policy runs before signing. The "✓ policy: allow" line is the M1 guarantee made visible.
  • Inline explorer button. One tap from chat into the real testnet — the demo's money shot.
  • No Approve/Deny yet. M2 ships every payment under the threshold. M3 wires the inline-keyboard Approve/Deny prompt for over-threshold payments.
  • Deterministic agent. The "results" are canned. Claude tool use lands at M3.

03

Commands

Five commands cover the entire M2 surface. M3 adds Approve/Deny inline buttons (they don't show as commands — they appear in-chat when a payment crosses the threshold).

/start

Welcome message + the current policy defaults. Run this first.

/task <query>

Kicks off the agent loop. Streams every event (probing, policy, signing, settled, results) as messages. End state: a real testnet tx with an explorer button.

/budget

Shows the current policy values (total / per-tx / daily caps, approval threshold, halt state, merchant address).

/halt

Kill switch — toggle. When halted, the bot refuses to start new tasks. M3 escalates this to a hard policy-engine refusal during in-flight runs.

/forget

Wipes your per-user session state. M3 extends this to also wipe your stored Anthropic API key (BYOK).

/explorer <hash>

Returns the testnet explorer URL for any tx hash. Handy for re-opening a previous run's proof.

04

What this demo does NOT do yet

Honest about the M2 scope so you can read the demo correctly. None of these are blockers; they're the M3 punch list.

M3 PUNCH LIST

① Claude reasoning — agent is deterministic in M2. M3 wires the Anthropic API (BYOK) so the agent thinks before it pays. M3

② Approve / Deny prompt — every payment is below threshold in M2, so the prompt never fires. M3 emits an inline-keyboard message when a payment crosses the threshold; you tap to release or refuse the signature. M3

③ Hard kill switch — /halt currently blocks new /task calls but doesn't interrupt an in-flight run. M3 routes the halt flag through the policy engine, refusing every future signature mid-run. M3

④ BYOK API key storage — bot doesn't ask for an Anthropic key in M2 (no Claude calls yet). M3 adds onboarding for a scoped+capped key, encrypted at rest. M3
