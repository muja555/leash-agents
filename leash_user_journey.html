User journey · screens, flow, and connections

Leash — what the user actually sees

Each screen in simple wireframes, the connections between them, and the path from task to verified report. v1 ships as a web app (localhost:8080); the same loop maps to mobile later.

Six screens. One happy path: S1 → S2 → S3 → S6. Two branches you can open any time: S4 (Tx detail) and S5 (Budget & policy). The agent does the work, the human stays in command.

STRATEGY · APPROACH A — CONTROL PLANE OVER THE RAILS
Leash is the neutral, non-custodial control tower for agents that pay — it sits above the payment protocols (x402 / AP2) and on top of wallet infra, owning the layer they treat as an afterthought: policy, human oversight, and operator UX. Three pillars are now scaffolded so going live is config + one adapter:
· Multi-chain — XRPL live; Solana / Base / Ethereum behind one PaymentAdapter interface, gated by the same chain-agnostic policy engine.
· AI tokens — many models via one gateway (OpenRouter), billed from prepaid USD credits or BYOK.
· Non-custodial — the user funds their own wallet; Leash never holds funds destined for merchants (sidesteps money-transmitter licensing).
Halt / Deny / Block Launch / Allow Money / Budget Navigate

THE END-TO-END GOAL
Question + budget  →  agent pays per use on XRPL (Solana / Base next)  →  sourced report with explorer-verifiable receipts.

FROM S3 · LIVE RUN — IN-PLACE CONTROLS
These don't navigate — they act on the run you're already watching.
APPROVE M3

A payment crossed the threshold. Approve it; the agent signs that one tx and the loop continues.

DENY M3

Deny the payment. The agent is told the resource is unavailable and re-plans. No signature was ever produced.

PAUSE / RESUME M2

Soft-stop after the current step. Resumable. Use it when you want to read the feed before more money moves.

KILL SWITCH M3

Available on S1 and S3. Sets the agent to halted. The policy engine refuses every signature from that moment. (Button exists from M2; actually halts in M3 when the policy engine gates real signing.)

04

The three moments that sell the demo

The journey has many screens, but it lands or doesn't on these three moments.

MOMENT 1 · S4 M1M2

The first tx hash on the explorer

A real, opened-in-a-browser XRPL testnet transaction tied to a payment the agent decided to make. Proof that this isn't a slideshow. The tx itself exists from M1 (terminal); the moment lands on a phone in M2.

MOMENT 2 · S3 M3

The Approve / Deny prompt

The phone vibrates mid-run because a payment crossed the threshold. The person taps once and the loop continues — or doesn't. Autonomy plus control, in one tap. This is the demo-video moment.

MOMENT 3 · S1 / S3 M3

The kill switch lands

One tap halts every future signature. Not a soft pause — the policy engine refuses to sign. This is the floor that makes trusting the rest of the loop possible.
